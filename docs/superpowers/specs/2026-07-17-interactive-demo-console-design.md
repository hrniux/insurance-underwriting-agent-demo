# 中文交互核保演示台设计方案

## 1. 背景

项目已经提供四组结构化假数据、核保评估接口、场景目录接口和中文演示文档，但首次体验仍然需要阅读文档、执行 `curl` 并理解 JSON 响应。对于面试演示、业务讲解和新成员学习，这条路径仍有一定门槛。

本轮增加一个零构建依赖的中文交互演示台，让使用者可以在浏览器中选择假数据场景、查看投保事实、运行核保 Agent，并按业务语言理解模型结论、规则命中、知识证据、工具调用和七步执行轨迹。

## 2. 目标与成功标准

### 2.1 目标用户

- 希望快速理解项目能力的面试官或评审人员；
- 需要讲解智能核保流程的业务人员；
- 第一次接触 Agent 编排、规则引擎和 RAG 的开发者；
- 需要验证四组假数据能否覆盖不同核保结论的测试人员。

### 2.2 成功标准

1. 应用启动后，访问 `/demo/` 即可进入中文演示台；
2. 页面能展示全部四组假数据，并明确标注“仅供演示”；
3. 选择场景后，无需阅读原始 JSON 就能理解保单、报价、历史、查勘和灾害信息；
4. 点击一次按钮即可运行真实核保接口，并展示实际结论与预期结论是否一致；
5. 页面能用中文解释风险分、模型摘要、原因、建议、规则命中、知识证据、工具调用和七步轨迹；
6. 页面在桌面和手机宽度下均可使用，键盘操作和焦点状态清晰；
7. 不增加 Node.js、React、Vue、图表库或外部字体等运行依赖；
8. 自动化测试、真实 HTTP 验证和浏览器验证均通过。

## 3. 方案比较与选择

### 3.1 方案 A：只补充中文文档和命令脚本

优点是改动最少，也不会增加前端代码。缺点是用户仍需要复制命令、阅读 JSON，不能直观看到 Agent 的处理过程，无法显著降低理解门槛。

### 3.2 方案 B：独立 React 或 Vue 前端

优点是组件生态成熟，后续扩展方便。缺点是需要额外的包管理、构建链和部署配置，会让一个面试型 Demo 承担不必要的工程复杂度，也增加离线运行风险。

### 3.3 方案 C：Spring Boot 原生静态资源演示台

采用语义化 HTML、原生 JavaScript 和 CSS，文件由 Spring Boot 直接提供。页面调用现有场景目录接口和核保评估接口，不新增业务接口。

本项目选择方案 C。它能明显改善演示体验，同时保持单体启动、无前端构建、无外部资源依赖，最符合“简单明了、便于面试讲解”的项目定位。

## 4. 总体架构

```text
浏览器 /demo/
  ├─ index.html：语义化页面骨架
  ├─ styles.css：响应式视觉和可访问状态
  └─ app.js：场景状态、接口调用和安全渲染
       ├─ GET  /api/v1/demo/scenarios
       ├─ GET  /api/v1/demo/scenarios/{policyNo}
       └─ POST /api/v1/underwriting/evaluations
```

演示台只是现有后端能力的只读展示层。场景选择只读取假数据，运行核保只创建一次评估记录，不提供假数据编辑、删除或生产数据接入功能。

## 5. 页面信息架构

### 5.1 顶部说明区

- 项目名称和一句话价值说明；
- “本页面使用虚构数据”状态标识；
- 后端服务连接状态；
- 指向 Swagger UI 和中文数据指南的辅助说明。

### 5.2 场景选择区

- 展示 P-1001、P-2001、P-3001、P-4001 四张场景卡片；
- 每张卡片包含场景名称、简要说明、预期结论和保额；
- 当前场景使用 `aria-pressed` 与视觉边框双重标识；
- 键盘用户可通过 Tab 和 Enter/Space 完成切换。

### 5.3 场景事实区

- 场景摘要、业务问题和学习要点；
- 保单信息：产品、地区、行业、保额和免赔额；
- 报价信息：保费、费率和报价状态；
- 历史信息：近三年赔付次数、赔付金额和主要事故；
- 查勘信息：查勘等级、整改状态和关键发现；
- 灾害信息：主要灾害、风险等级和风险描述。

字段以业务标签和格式化值展示，避免要求用户理解接口字段名。

### 5.4 核保操作区

- “运行智能核保”作为页面主操作；
- 请求期间禁用按钮并显示明确进度；
- 状态变化通过 `aria-live` 向辅助技术播报；
- 场景切换时忽略已过期的网络响应，避免旧结果覆盖新场景。

### 5.5 核保结果区

结果按“先结论、后依据、再过程”的顺序展示：

1. 实际核保结论、预期结论和一致性提示；
2. 风险等级、风险分数和可读进度条；
3. 模型摘要、核保原因和建议动作；
4. 规则命中卡片，展示规则编码、严重度、分数影响和原因；
5. 知识证据卡片，展示标题、类型、相关度和摘要；
6. 七步 Agent 时间线，展示每一步的状态、耗时和错误码；
7. 工具调用记录，展示工具名称、状态、耗时、输入摘要和输出摘要。

颜色只作为辅助表达。所有结论、状态和风险等级都同时提供中文文字。

## 6. 文件职责

### 6.1 `src/main/resources/static/demo/index.html`

- 只包含稳定、语义化的页面结构；
- 使用 `header`、`main`、`section`、`button`、`dl` 等合适元素；
- 为动态区域提供固定容器、标题和可访问说明；
- 不内嵌大段脚本或样式，保持文件职责清晰。

### 6.2 `src/main/resources/static/demo/app.js`

- 管理当前场景、场景详情、核保结果、加载状态和错误状态；
- 封装 `requestJson`，统一检查 HTTP 状态和解析 Problem Details；
- 使用 `document.createElement`、`textContent` 和 `replaceChildren` 渲染接口数据；
- 禁止把接口返回值拼接到 `innerHTML`，避免演示数据变化时引入脚本注入风险；
- 对分数、耗时和数组字段做类型检查与默认值处理；
- 使用请求序号或 `AbortController` 避免竞态覆盖；
- 将英文枚举映射为中文业务标签，同时保留原始编码便于技术讲解。

### 6.3 `src/main/resources/static/demo/styles.css`

- 使用本地系统字体和 CSS 自定义属性，不请求外部资源；
- 采用稳重的深蓝、青绿色主色和暖灰背景，突出保险业务的可靠感；
- 为通过、人工复核和拒保提供不同徽标，并保留文字标签；
- 桌面宽度使用场景侧栏与内容区布局，窄屏自动改为单列；
- 提供清晰的 `:focus-visible`、加载、错误、空状态和禁用状态；
- 尊重 `prefers-reduced-motion`，减少不必要动画。

## 7. 数据流与状态管理

### 7.1 首次加载

1. 请求场景目录；
2. 渲染场景卡片；
3. 默认选择第一组场景；
4. 请求该场景详情并渲染事实区；
5. 核保结果区保持“尚未运行”状态。

### 7.2 切换场景

1. 更新选中状态并清除旧核保结果；
2. 禁用核保按钮，显示详情加载状态；
3. 请求新场景详情；
4. 只有当前请求仍有效时才更新页面；
5. 加载完成后恢复核保按钮。

### 7.3 运行核保

1. 使用场景详情中的 `policyNo` 和 `question` 构造请求；
2. 调用真实 `/api/v1/underwriting/evaluations` 接口；
3. 请求期间阻止重复提交；
4. 成功后渲染结论、依据和过程；
5. 根据 `expectedResult.decision` 比较实际结果，显示一致性提示；
6. 失败时保留当前场景，并提供可读错误和重试入口。

## 8. 错误处理与安全边界

- 网络失败、非 2xx 响应和非法 JSON 使用统一中文错误面板；
- 如果后端返回 Problem Details，优先展示 `title`、`detail` 和 `traceId`；
- 错误信息不覆盖场景事实，用户可以直接重试；
- 动态文本一律通过安全 DOM API 写入；
- 风险分数在展示前限制到 0—100，避免异常值破坏布局；
- 外部链接使用安全属性，页面不执行第三方脚本；
- 页面显著说明全部数据均为虚构内容，不可用于真实承保判断；
- 本轮不引入登录、授权和生产部署承诺。

## 9. 可访问性与响应式要求

- 所有交互元素使用原生按钮或链接；
- 动态状态区域设置 `aria-live="polite"`，严重错误使用 `role="alert"`；
- 当前场景同时使用 `aria-pressed` 和可见文字/边框标识；
- 风险进度条提供 `role="progressbar"`、当前值和文本说明；
- 页面标题层级连续，列表使用真实列表语义；
- 正文和状态文字满足可读对比度；
- 在约 360 像素宽度下不产生关键内容横向滚动；
- 工具调用信息在窄屏下从表格式布局切换为纵向卡片。

## 10. 测试策略

### 10.1 自动化测试

新增静态资源契约测试，验证：

- `/demo/` 能返回演示台 HTML；
- JavaScript 和 CSS 静态资源可以访问且类型正确；
- HTML 包含关键可访问区域和核保操作入口；
- JavaScript 使用正确的场景和评估接口路径；
- 中文文档包含演示台访问地址和使用说明。

测试按照“先失败、再实现、再通过”的顺序编写。

### 10.2 真实运行验证

- 执行 `mvn clean verify`；
- 启动打包后的应用并通过真实 HTTP 请求检查三个静态资源；
- 在真实浏览器中完成场景加载、场景切换和核保运行；
- 至少验证一组自动通过场景和一组拒保场景；
- 检查桌面与手机宽度布局；
- 检查浏览器控制台没有运行时错误。

## 12. 非目标

本轮明确不做以下内容：

- 不引入 React、Vue、npm 或独立前端工程；
- 不提供假数据在线编辑、上传或删除能力；
- 不实现用户登录、权限管理或生产级安全体系；
- 不增加新的业务接口或修改核保决策逻辑；
- 不引入图表库、外部字体、CDN 或在线图片；
- 不把演示台描述为真实生产核保工作台。
