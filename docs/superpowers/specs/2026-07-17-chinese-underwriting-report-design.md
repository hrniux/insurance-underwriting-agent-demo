# 中文核保 Markdown 报告设计

日期：2026-07-17  
状态：已确认（依据用户“全权处理、无需再次确认”的授权自主选型）

## 1. 背景与目标

当前项目已经可以在浏览器中运行四组虚构场景，并查看结论、规则、知识证据、七步 Agent 轨迹和工具调用记录。下一步需要把一次真实评估沉淀成便于保存、分享和面试讲解的中文材料。

本次新增“中文核保 Markdown 报告”：用户完成单场景核保后，可以一键下载由后端根据已保存评估结果生成的 `.md` 文件。报告只重组现有结果，不重新执行核保，也不引入第二套业务判断。

## 2. 方案比较

### 方案 A：后端生成 Markdown，前端提供下载入口（采用）

- 优点：报告口径集中、可通过接口测试、可被浏览器和其他客户端复用；
- 优点：直接读取已持久化评估，不会重复调用模型或规则；
- 优点：Markdown 轻量、可读、适合代码仓库和面试展示；
- 代价：需要维护一组中文枚举标签和 Markdown 转义逻辑。

### 方案 B：浏览器使用 JavaScript 拼接 Markdown

- 优点：无需新增后端接口；
- 缺点：中文标签、字段顺序和格式逻辑会散落到前端，难以复用和做完整接口验收。

### 方案 C：后端直接生成 PDF

- 优点：打印效果稳定；
- 缺点：需要字体、分页和渲染依赖，明显超出当前“简单明了、便于理解”的 Demo 定位。

因此采用方案 A，并把 PDF 作为未来可选演进方向。

## 3. 用户流程

1. 用户在 `/demo/` 选择一组虚构场景并点击“运行智能核保”；
2. 浏览器调用现有 `POST /api/v1/underwriting/evaluations`，展示真实返回结果；
3. 结果区显示“下载中文 Markdown 报告”链接；
4. 链接访问 `GET /api/v1/underwriting/evaluations/{evaluationId}/report`；
5. 后端读取已保存的评估结果并返回 UTF-8 Markdown 附件；
6. 切换场景或重新运行前，旧下载链接会被清理，避免下载错报告。

## 4. API 设计

### 请求

```http
GET /api/v1/underwriting/evaluations/{evaluationId}/report
Accept: text/markdown
```

### 成功响应

- 状态码：`200 OK`；
- `Content-Type: text/markdown;charset=UTF-8`；
- `Content-Disposition: attachment; filename="underwriting-report-{evaluationId}.md"`；
- 响应体：中文 Markdown 文本。

### 失败响应

评估不存在时，继续复用项目统一的 RFC 9457 Problem Details 响应和 `EVALUATION_NOT_FOUND` 错误码，不增加特殊异常格式。

## 5. 报告结构

报告固定包含以下内容：

1. 标题、生成说明与虚构数据边界；
2. 评估基本信息：评估编号、会话编号、保单号、核保问题、评估时间；
3. 核保结论：中文结论和原始枚举、中文风险等级和原始枚举、风险分；
4. 模型摘要、核保原因、建议动作；
5. 规则命中表：规则编码、严重程度、决策、风险分影响、原因；
6. 知识证据表：文档类型、标题、相关度、摘录、文档和分块编号；
7. 七步 Agent 执行轨迹：步骤、状态、耗时、开始时间、错误码；
8. 工具调用表：工具、状态、耗时、输入摘要、输出摘要、错误码；
9. 模型执行信息：提供方、模型、尝试次数、是否降级；
10. 免责声明。

所有枚举展示为“中文标签（`ORIGINAL_CODE`）”，既方便初学者阅读，也方便面试时定位代码。空列表统一显示“无”，不会生成空白章节。

## 6. 格式与安全约束

- 表格单元格中的反斜杠、竖线和换行必须转义，防止用户问题或模型文本破坏 Markdown 表格；
- 列表文本中的换行折叠为 `<br>`，保留可读性；
- 文件名只使用系统生成的评估编号，不使用保单号、被保险人或用户问题；
- 报告不包含 API Key、完整提示词或工具原始对象，只输出评估响应中已经公开的审计摘要；
- 本功能只导出已有结果，不重新执行核保，因此下载操作无业务副作用。

## 7. 代码边界

- 新增 `report/UnderwritingMarkdownReportService`：只负责确定性格式化；
- `UnderwritingEvaluationController`：负责读取评估、设置 HTTP 响应头并返回报告；
- `demo/index.html`、`app.js`、`styles.css`：负责显示、更新和清理下载入口；
- README、API 示例、虚构数据指南和面试指南：说明使用方法、数据边界与讲解重点。

报告服务不依赖 Web API 类型，输入直接使用领域对象 `UnderwritingEvaluation`，便于单元测试和未来从批处理、邮件或对象存储场景复用。

## 8. 验收标准

- 报告服务单元测试证明核心章节、中文标签、空值展示和 Markdown 转义正确；
- API 集成测试证明响应状态、媒体类型、下载文件名和真实 `P-1001` 内容正确；
- 不存在的评估继续返回统一 404 Problem Details；
- 静态资源契约测试证明下载入口和安全 DOM 更新存在；
- 中文文档覆盖浏览器下载和 curl 下载两种路径；
- `mvn clean verify`、真实 JAR 接口冒烟和浏览器桌面/移动端验收通过；
- 最终提交合并并推送到 GitHub `main`，本地与远端 SHA 一致。
