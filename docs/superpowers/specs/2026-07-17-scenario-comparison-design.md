# 全场景横向对比设计

## 1. 背景与目标

现有中文智能核保演示台适合逐个理解 `P-1001` 至 `P-4001`，但面试演示者仍需要手工切换四次，才能说明为什么低风险场景自动通过、中风险场景转人工、高风险底线场景拒保。

本轮增加“一键对比全部场景”能力。用户在同一个页面内运行四组虚构场景，每一组都调用现有 `POST /api/v1/underwriting/evaluations`，随后横向查看实际结论、预期结论、风险分、规则命中数、知识证据数和完整耗时。这个能力用于教学与面试展示，不新增核保规则，也不改变现有单场景流程。

## 2. 方案比较与选择

### 方案 A：浏览器顺序调用现有评估 API（采用）

浏览器读取已经加载的场景目录，按目录顺序逐个调用现有评估 API，并在全部请求结束后生成对比摘要和场景卡片。

- 优点：四组结果都来自真实七步 Agent；不新增批量业务接口；不复制规则或预期结果；可以展示实时进度和部分成功。
- 代价：默认 Mock 模式需要发出四次请求；外部模型模式下总耗时会高于并行调用。
- 选择理由：演示项目更重视过程可解释、请求稳定和代码复用，顺序执行还能降低外部模型限流风险。

### 方案 B：增加后端批量评估接口

后端接收多个场景并聚合结果，浏览器只发出一个请求。

- 优点：前端逻辑更少，可以在服务端统一批量策略。
- 代价：引入只服务演示页面的新业务接口；需要额外定义部分失败协议；容易让批量编排与单次编排形成两套维护入口。

### 方案 C：展示预计算静态对比

直接用场景文件中的 `expectedResult` 生成对比结果。

- 优点：速度最快，不产生评估记录。
- 代价：无法证明真实规则、RAG、模型和轨迹被执行；修改业务逻辑后可能继续展示旧预期。

## 3. 用户体验

场景选择区增加“对比全部场景”按钮和独立的 `aria-live` 状态。场景目录加载完成前按钮不可用；运行期间按钮显示“正在对比 2/4……”并保持禁用；结束后恢复为“重新对比全部场景”。

对比结果作为第四步展示，包含两个层次：

1. 总览摘要：成功场景数、结论分布、预期一致数、风险分范围、规则命中总数和知识证据总数。
2. 场景卡片：保单号、场景名称、实际结论、预期结论、是否一致、风险等级、风险分、规则数、证据数和评估耗时。

每张成功卡片提供“查看单场景详情”按钮。点击后沿用现有 `selectScenario`，切换到对应虚构场景并滚动到投保事实区域；用户仍需主动点击“运行智能核保”才能生成该场景的详细结果，避免把批量结果悄悄写入单场景状态。

失败卡片展示场景名称和安全的中文错误信息，不阻止其他场景继续运行。若四组全部失败，对比区仍展示四张失败卡并在状态中明确成功数为零。

## 4. 组件与数据边界

### `index.html`

新增批量对比按钮、进度状态和第四步结果容器。所有交互元素使用原生按钮和语义化区域，不引入第三方组件。

### `app.js`

继续作为页面控制器，新增以下职责：

- 保存独立的 `comparisonController`，防止重复运行；
- 顺序调用现有评估 API；
- 将成功和失败规范化为页面内部对比项；
- 计算聚合摘要并安全渲染卡片；
- 在查看详情时复用现有场景选择流程。

批量对比不会修改 `state.evaluation`、`state.detail` 或单场景结果区域，因此两种演示模式相互独立。

### `styles.css`

新增对比操作区、摘要网格、场景卡片、紧凑风险条和失败态样式。在宽屏上使用自适应多列卡片，在 760 像素和 470 像素断点下退化为单列，不引入横向滚动。

### 后端与场景数据

不修改后端 API、规则引擎和场景 JSON。浏览器使用场景目录中已有的 `policyNo`、`question` 和 `expectedResult` 组装现有评估请求。实际结论始终来自 `UnderwritingAgentOrchestrator`。

## 5. 数据流

1. 页面通过 `GET /api/v1/demo/scenarios` 取得四组场景摘要。
2. 用户点击“对比全部场景”。
3. 浏览器按场景目录顺序逐个发送 `POST /api/v1/underwriting/evaluations`。
4. 每次成功后保存真实评估结果；每次失败后保存安全错误信息并继续下一组。
5. 浏览器计算成功数、结论分布、预期一致数、风险分范围、规则总数和证据总数。
6. 页面使用 `textContent`、`replaceChildren` 和 DOM 节点 API 渲染总览与场景卡片。

## 6. 状态与错误处理

- 场景目录为空时沿用现有初始化错误，不允许批量运行。
- 运行期间拒绝第二次批量操作，但不禁用单场景选择。
- 批量请求使用独立 `AbortController`，不与单场景详情或单场景评估互相取消。
- 一个请求失败后继续运行其余场景，并在失败卡片显示服务返回的安全错误详情。
- 页面卸载时由浏览器终止未完成请求；本轮不增加额外取消按钮。
- 对比结果只存在于当前页面内，刷新后清空；后端评估记录仍遵循现有内存仓库行为。

## 7. 可访问性与安全

- 对比按钮使用原生 `<button type="button">`。
- 进度状态使用 `aria-live="polite"`，错误汇总使用可读文本，不只依赖颜色。
- 风险条使用 `role="progressbar"` 以及 `aria-valuemin`、`aria-valuemax`、`aria-valuenow`。
- 查看详情按钮名称包含场景名称，便于读屏区分。
- 所有服务端文本通过 `textContent` 或节点构造写入，不使用 `innerHTML`。
- 不增加外部脚本、字体、图片、分析服务或网络依赖。

## 8. 测试与验收

### 自动化测试

- 先扩展 `DemoConsoleStaticResourceTest`，要求 HTML 包含批量按钮、进度状态和结果容器；测试在实现前应失败。
- 要求客户端包含顺序对比入口、独立 `AbortController`、聚合摘要、部分失败处理和查看详情行为，同时继续禁止 `innerHTML`。
- 要求样式包含对比摘要、对比卡片、失败态和移动端规则。
- 更新 `DocumentationContractTest`，要求 README、假数据指南和面试指南包含全场景对比说明。
- 运行 `mvn clean verify`，确保全部 Java 和 Spring 集成测试通过。

### 真实运行验收

- 启动打包后的应用，确认健康检查、演示页面及静态资源正常。
- 在浏览器点击“对比全部场景”，确认四组场景均完成，结论分布为自动通过 1、人工复核 2、拒保 1，且四组实际结论都与预期一致。
- 确认风险分范围为 10–70，四张卡片均能进入对应单场景详情。
- 在 390 像素宽度验证无页面级横向溢出、按钮可操作、状态可读。
- 检查浏览器控制台无脚本错误。

## 9. 明确不包含的范围

- 不新增真实保险数据、生产规则或真实承保建议。
- 不增加后端批量评估 API。
- 不增加结果导出、持久化、登录、权限和多租户。
- 不更换前端技术栈，不引入 npm、框架或第三方图表库。
- 不改变已有单场景核保结论和七步编排逻辑。
