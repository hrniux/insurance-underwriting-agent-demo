# 虚构核保数据与十分钟学习指南

## 1. 文档目的

本文用四组可对照的虚构保单，解释财险智能核保 Agent 如何从业务资料得到风险分和核保结论。阅读时不需要预先了解保险术语，也不需要配置外部模型或数据库。

你将看到同一条处理链路如何产生四种有代表性的结果：自动通过、人工复核、中风险边界和拒保。

## 2. 适用对象

- 第一次接触财产险核保的 Java 学习者；
- 希望理解 Agent、RAG、规则引擎和 MCP 如何协作的开发者；
- 需要在面试中演示可解释核保流程的候选人；
- 准备修改示例数据或接入真实内部系统的维护者。

如果你只想运行一次完整演示，可以直接阅读下一节；如果你想理解代码，再继续阅读场景推导和字段字典。

## 3. 十分钟学习路线

### 3.1 浏览器交互演示（推荐）

应用启动后打开 <http://localhost:8080/demo/>。这是理解项目最快的入口，不需要复制命令或直接阅读 JSON。

1. 在“第一步：选择演示场景”中选择 `P-1001` 至 `P-4001`；
2. 在“第二步：理解投保事实”中对照保单、报价、历史、查勘和灾害五组信息；
3. 点击“运行智能核保”，在“第三步”按实际结论与预期结论、风险分、核保原因、建议动作、规则命中、知识证据、七步 Agent 轨迹、工具调用记录的顺序阅读结果；
4. 在“人工复核反馈闭环”中选择核保人的处理结论并填写说明，观察 Agent 建议和人工结论如何分别保留；
5. 点击“下载中文 Markdown 报告”，把包含复核记录的结果保存为可独立阅读的文件。

建议先运行 `P-2001`，观察无规则命中时如何自动通过；再运行 `P-4001`，观察重大消防缺陷如何形成不可被模型降低的拒保底线。页面切换场景时会清空旧结果，接口失败时会保留当前事实并显示中文错误，可直接再次点击按钮重试。

页面展示的仍是 `underwriting-scenarios.json` 中的虚构数据，并调用真实 `/api/v1/underwriting/evaluations` 接口。它不是预先写死的结果截图，也不构成真实保险核保意见。

#### 全场景横向对比

点击“对比全部场景”后，页面按 `P-1001` 至 `P-4001` 的目录顺序调用四次真实核保接口。建议按以下顺序阅读：

1. 先看结论分布，确认自动通过 1、人工复核 2、拒保 1；
2. 再看风险分范围 10–70，以及每组规则命中数和知识证据数；
3. 最后点击“查看单场景详情”，回到五类业务事实解释差异来源。

如果某组请求失败，其他场景仍会继续运行。部分场景的失败卡片表示技术请求未完成，不表示业务结论为拒保，也不能作为真实承保判断。

#### P-1001 报告阅读顺序

运行 `P-1001` 后下载报告，可以按下面的因果链阅读这组假数据：

1. “基本信息”确认报告属于虚构保单 `P-1001`，并记录评估编号与生成时间；
2. “核保结论”给出人工复核（`MANUAL_REVIEW`）、高风险（`HIGH`）和 70 分；
3. “人工复核闭环”把 Agent 辅助建议与核保人的最终处理并列展示，原始建议不会被覆盖；
4. “规则命中”解释 70 分如何由 10 分基础分、红色暴雨、重复出险、高保额和未完成整改叠加而来；
5. “知识证据”给出条款、规则、指引或历史案例的文档与分块编号，说明结论依据来自哪里；
6. “七步 Agent 执行轨迹”和“工具调用记录”回答系统执行了什么、是否成功、分别耗时多久；
7. “模型执行信息”明确实际 provider、model、尝试次数以及是否发生降级。

报告接口只是把当前评估仓库中的领域对象确定性格式化为 Markdown，不会重新执行核保、重新调用模型或改变结论。切换场景后页面会隐藏旧链接；必须先完成新的单场景评估，才能下载对应报告。

## 4. 四组虚构场景总览

| 保单号 | 场景 | 关键事实 | 预期决策 | 风险等级与分数 |
|---|---|---|---|---|
| `P-1001` | 临港高风险物流仓库 | 2,000 万元保额、2 次出险、排水整改未完成、暴雨红色 | 人工复核（`MANUAL_REVIEW`） | 高风险（`HIGH`），70 分 |
| `P-2001` | 低风险科技办公楼 | 500 万元保额、无出险、无待整改事项、灾害风险低 | 自动通过（`APPROVE`） | 低风险（`LOW`），10 分 |
| `P-3001` | 暴雨暴露商贸仓库 | 1,200 万元保额、1 次出险、整改完成、暴雨红色 | 人工复核（`MANUAL_REVIEW`） | 中风险（`MEDIUM`），40 分 |
| `P-4001` | 极端火灾风险制造厂房 | 800 万元保额、消防重大缺陷、火灾风险极端 | 拒保（`REJECT`） | 高风险（`HIGH`），60 分 |

这些公司、地址、保单、金额、查勘结论和灾害等级全部是虚构数据。

## 5. 场景逐步推导

### 5.1 统一计分方法

所有场景从 10 分开始。命中规则后累加分数，并取规则中最严格的决策：

| 规则代码 | 触发条件 | 加分 | 决策下限 |
|---|---|---:|---|
| `CRITICAL_FIRE_DEFECT` | 火灾风险为极端，且消防状态为重大缺陷或缺失 | +50 | 拒保（`REJECT`） |
| `RED_RAINSTORM` | 暴雨风险达到红色或更高 | +20 | 人工复核（`MANUAL_REVIEW`） |
| `REPEATED_CLAIMS` | 近三年出险次数不少于 2 次 | +15 | 人工复核（`MANUAL_REVIEW`） |
| `HIGH_SUM_INSURED` | 保额不少于 1,000 万元 | +10 | 人工复核（`MANUAL_REVIEW`） |
| `OPEN_REMEDIATION` | 排水整改尚未完成 | +15 | 人工复核（`MANUAL_REVIEW`） |

风险等级区间：

- 0–29 分：低风险（`LOW`）；
- 30–59 分：中风险（`MEDIUM`）；
- 60–79 分：高风险（`HIGH`）；
- 80–100 分：极高风险（`CRITICAL`）。

决策严格程度为：自动通过（`APPROVE`）< 人工复核（`MANUAL_REVIEW`）< 拒保（`REJECT`）。模型只能解释结果，不能降低规则给出的决策下限。

### 5.2 `P-1001`：多项风险叠加

```text
基础分 10
+ 红色暴雨 20
+ 近三年重复出险 15
+ 保额达到 1,000 万元 10
+ 排水整改未完成 15
= 70 分，高风险，人工复核
```

这组场景说明：多个看似独立的业务事实会叠加，Agent 需要保留每条规则的命中原因，而不是只输出一个结论。

### 5.3 `P-2001`：证据齐全时自动通过

```text
基础分 10
+ 无规则命中 0
= 10 分，低风险，自动通过
```

如果 RAG 没有检索到任何知识证据，编排器仍会把最终决策提高到人工复核，避免在没有依据时自动承保。默认演示知识库包含相关证据，因此该场景可以展示自动通过。

### 5.4 `P-3001`：中风险边界

```text
基础分 10
+ 红色暴雨 20
+ 保额达到 1,000 万元 10
= 40 分，中风险，人工复核
```

近三年只有 1 次出险，不满足“2 次及以上”；排水整改已经完成，也不会命中整改规则。这个场景适合解释“风险等级”和“核保决策”不是同一个概念：中风险仍可能因为特定规则而必须人工复核。

### 5.5 `P-4001`：拒保底线

```text
基础分 10
+ 极端火灾风险叠加消防重大缺陷 50
= 60 分，高风险，拒保
```

这张保单的 800 万元保额没有触发高保额规则，但消防重大缺陷与极端火灾暴露的组合已经触发拒保（`REJECT`）。即使模型生成了更宽松的文字建议，最终决策也不能被降级。

## 6. 字段字典

场景文件位于 `src/main/resources/demo/underwriting-scenarios.json`。每个顶层对象代表一张虚构保单。

### 6.1 教学元数据

| 字段 | 含义 | 示例 |
|---|---|---|
| `policyNo` | 场景唯一保单号，也是五类事实的关联键 | `P-3001` |
| `name` | 中文场景名称 | 暴雨暴露商贸仓库 |
| `summary` | 一句话说明该场景的教学目的 | 展示人工复核与中风险的边界案例 |
| `question` | 推荐提交给核保评估接口的问题 | 这张保单为什么需要人工复核？ |
| `learningPoints` | 建议关注的知识点列表 | 区分风险等级与核保决策 |
| `expectedResult` | 按当前确定性规则计算的预期结果 | 40 分、中风险、人工复核 |

### 6.2 保单 `policy`

| 字段 | 含义 |
|---|---|
| `productCode` | 险种代码；本 Demo 使用企财险 `PROPERTY` |
| `insuredName` | 虚构投保企业名称 |
| `occupancy` | 标的用途，例如仓库 `WAREHOUSE`、办公楼 `OFFICE`、厂房 `FACTORY` |
| `address` | 虚构标的地址 |
| `startDate` / `endDate` | 保险起期与终期，终期必须晚于起期 |

### 6.3 报价 `quotation`

| 字段 | 含义 | 单位 |
|---|---|---|
| `sumInsured` | 保险金额，即本次承保的金额上限 | 人民币元 |
| `rate` | 示例费率 | 小数，例如 `0.0035` |
| `premium` | 示例保费 | 人民币元 |
| `deductible` | 免赔额，即损失中由被保险人自行承担的部分 | 人民币元 |

### 6.4 历史核保 `history`

| 字段 | 含义 |
|---|---|
| `claimCountThreeYears` | 近三年出险次数 |
| `paidLossThreeYears` | 近三年已赔金额，单位为人民币元 |
| `priorDecisions` | 历史核保结论或附加条件 |

### 6.5 风险查勘 `survey`

| 字段 | 含义 |
|---|---|
| `fireProtectionStatus` | 消防设施状态；`CRITICAL_DEFECT` 表示重大缺陷 |
| `drainageRemediationCompleted` | 排水整改是否完成 |
| `openIssues` | 尚未关闭的问题列表 |
| `conclusion` | 查勘人员的文字结论 |

### 6.6 灾害风险 `disaster`

| 字段 | 含义 |
|---|---|
| `riskZone` | 虚构灾害风险区域 |
| `rainstorm` | 暴雨风险等级 |
| `flood` | 洪水风险等级 |
| `fire` | 火灾风险等级 |
| `dataDate` | 风险数据评估日期 |

## 7. 枚举对照

### 7.1 核保决策 `Decision`

| 英文值 | 中文含义 |
|---|---|
| `APPROVE` | 自动通过 |
| `MANUAL_REVIEW` | 人工复核 |
| `REJECT` | 拒保 |

### 7.2 风险等级 `RiskLevel`

| 英文值 | 中文含义 |
|---|---|
| `LOW` | 低风险 |
| `MEDIUM` | 中风险 |
| `HIGH` | 高风险 |
| `CRITICAL` | 极高风险 |

### 7.3 灾害等级 `HazardLevel`

`UNKNOWN` 表示数据源不可用或无法确认，不等于低风险。已知等级从低到高依次为：低（`LOW`）、中（`MEDIUM`）、高（`HIGH`）、红色（`RED`）、极端（`EXTREME`）。规则会按照已知等级判断“达到红色或更高”；出现 `UNKNOWN` 时由安全降级策略强制转人工补充核验。

### 7.4 安全降级状态

- `ToolCallStatus.FAILED`：灾害工具本次调用实际失败；
- `StepStatus.DEGRADED`：资料采集虽然不完整，但在人工复核底线下允许继续；
- `DegradationNotice`：记录告警编码、工具、错误码、处置说明和决策下限。

可运行 `SPRING_PROFILES_ACTIVE=degraded-demo mvn spring-boot:run`，再执行 `P-2001`。正常场景的
10 分低风险会保留，但原本的自动通过将提升为人工复核，页面和下载报告都会显示降级告警。

### 7.5 人工复核结果 `HumanReviewOutcome`

| 英文值 | 中文含义 |
|---|---|
| `APPROVED` | 核保人同意承保 |
| `REJECTED` | 核保人拒绝承保 |
| `MORE_INFORMATION_REQUIRED` | 核保人要求继续补充资料 |

### 7.6 Agent 建议与人工结果关系 `AgentReviewRelationship`

| 英文值 | 中文含义 |
|---|---|
| `CONFIRMED` | 人工确认自动通过或拒保建议 |
| `OVERRIDDEN` | 人工推翻自动通过或拒保建议 |
| `RESOLVED_MANUAL_REVIEW` | 原建议为人工复核，核保人已给出承保或拒保结论 |
| `CONTINUED_MANUAL_REVIEW` | 原建议为人工复核，核保人仍要求补充资料 |

一条评估只能创建一条复核记录。这个 Demo 用不可覆盖记录强调审计语义；真实系统通常会把更正建模为
新的追加事件，并记录更正原因、操作者权限和电子签名，而不是直接更新旧数据。

## 8. 如何修改或新增场景

1. 打开 `src/main/resources/demo/underwriting-scenarios.json`。
2. 复制一整个场景对象，并使用新的顶层 `policyNo`。
3. 把 `policy`、`quotation`、`history`、`survey`、`disaster` 中的 `policyNo` 全部改成同一个值。
4. 确保所有金额和次数不为负数，保险终期晚于起期。
5. 根据第 5.1 节重新计算 `expectedResult`，规则代码不能重复。
6. 执行：

```bash
mvn -Dtest=DemoScenarioRepositoryTest,UnderwritingRuleEngineTest test
```

预期结果：数据加载和规则测试通过。如果保单号重复、金额为负数或日期错误，应用与测试会明确指出无效场景。

新增场景后，还应为它补充精确的规则测试和本文场景说明，避免数据、代码与文档不一致。

## 9. 常见问题

### 为什么场景目录显示的是“预期结果”？

场景目录展示由确定性规则推导的教学预期，便于运行前理解案例。真正的核保评估还会执行知识检索、模型建议和结果持久化；知识证据缺失时，最终决策可能被提高到人工复核，但不会被降低。

### 风险等级高就一定拒保吗？

不一定。风险等级由分数区间决定，决策由命中规则的最严格下限决定。`P-1001` 是高风险但需要人工复核；`P-4001` 是高风险且拒保。

### 修改 JSON 后为什么应用启动失败？

场景仓库会在启动时做完整性校验。请检查重复保单号、五类子对象的保单号、负金额、日期顺序、风险分范围和重复规则代码。

### 场景接口会执行核保或保存数据吗？

不会。`GET /api/v1/demo/scenarios` 和详情接口只读取教学数据。只有调用 `POST /api/v1/underwriting/evaluations` 才会运行七步 Agent 流程并保存评估结果。

### 下载报告会再执行一次核保吗？

不会。`GET /api/v1/underwriting/evaluations/{evaluationId}/report` 只读取指定的已保存评估及可选人工复核记录并格式化内容。它没有业务副作用。默认 Profile 重启后内存评估会丢失并返回 `EVALUATION_NOT_FOUND`；启用 `persistent-demo` 时，会话、评估和人工复核保存在 H2 文件中，可在进程重启后继续读取。

### 为什么人工复核不能再次覆盖？

覆盖会让人无法判断 Agent 当时给了什么建议、谁在什么时候作出了什么处理。Demo 对同一评估使用原子
`create`，第二次提交返回 `HUMAN_REVIEW_ALREADY_EXISTS`。生产系统如果允许更正，应新增带前序版本和
更正原因的审计事件，而不是修改原记录。

## 10. 虚构数据与非业务建议声明

本文及项目中的公司、地址、保单号、金额、费率、赔付、查勘结论、灾害等级、规则阈值和保险条款均为虚构数据，仅用于技术学习、软件测试和面试演示。

本项目不连接真实保险系统，不包含真实客户信息，输出不构成承保、拒保、定价、理赔或风险管理建议。真实业务必须由具备权限的核保人员依据公司制度、监管要求和完整资料作出决定。
